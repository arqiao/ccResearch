标题: Skill创作手记：我把东野圭吾的小说变成了无限流【侦探游戏】
日期: 20260207
原文: https://mp.weixin.qq.com/s/JSHFRlzDsplX9z3Bl8MgcQ

 id="js_content" style="visibility: hidden; opacity: 0; "> Part 1: 起源 —— 只是因为看了一眼那个 Demo 那天接触 PDF2Skills 的时候，我压根没想过自己会动手做一个侦探游戏。 起因特简单。我看到有人用这个项目跑了一个 Demo:5 个嫌疑人，3 次对话机会，最后猜凶手。 刚开始玩的时候，我觉得这就是个普通的“找茬”游戏。 但玩着玩着，我发现不一样。 手里拿到的线索卡，不是那种写死的剧情文本，而是真的需要你去分析、去推理、去组合的工具。 这背后的机制，是从 PDF 里提取出的 Skills 做出来的项目。 它把书里的死知识，变成了可玩的互动游戏。 以前看侦探小说，我是在脑子里“推理”；但这个游戏，让我觉得我是在手上“操作”。 我就在想：既然别人能做，我是不是也能复刻一个？ 我可是东野圭吾的铁粉啊。如果能把他书里的那些诡计、手法提取出来，做成游戏，那得多带感？ 我与其说是想“做游戏”，不如说是想“验证”：这个思路能不能复用？skills 到底能玩出什么花样？ Part 2: 念头 —— 不做游戏，做个实验 说实话，当时的想法特别模糊。 我并不是抱着“我要做一款超牛的小游戏”的决心开始的，更像是一种单纯的好奇—— 这事儿，到底能不能跑通？ 我想验证几个问题： 从小说里提炼 Skills, AI 能自己组合出意思的游戏吗？ 提炼出来的东西，怎么设计成可玩的游戏机制？ 这套流程跑下来，大概要花多长时间？ 更重要的是，我在赌一个可能性：如果这个流程是通的，那以后不管是技术文档、法律条文，还是晦涩的说明书，我是不是也能用类似的方式，提取 skills，然后做一些有意思的游戏或工具？ 所以，与其说我在 “做项目” ，不如说我在 “做实验” 。 没有 KPI，没有产品文档，甚至连选哪本书都没想好。我只是想试试看，看看这个思路能不能走通，看看能不能在这个过程中获得一些新的经验。 那时候的心态，与其说是“创作者”，不如说是“探索者”。 Part 3: 拆解 —— 怎么把书变成能跑的代码？ 动工前，得先搞清楚两件事：手里的铲子怎么用（工具），地基怎么挖（拆解）。 PDF2Skills 到底是啥？ 简单说，它就是个“知识榨汁机”。把 PDF 扔进去，它能把里面的知识点榨出来，变成 AI 能调用的“技能”。我试了两个路子： 第一种是用 Claude Code 硬提。我给 cc 东野圭吾的 整本「拉普拉斯的魔女」，让它帮我识别哪些是“可玩的技能”——比如“密室作案手法”、“不在场证明破解”这类。Claude Code 更灵活，可以根据我的需求调整提取的颗粒度。 第二种是用现成的项目PDF2SKills。已经有人搭建好了从 PDF 到 skills 的标准化流程，我直接把小说文档传进去等解构skills。这就是侦探游戏 demo 的那家提供的实验项目，不知道他们背后提取 skills 的提示词是什么，但值得一试（文末有介绍链接）。 项目怎么拆？ 我没敢想太远，先把大象装进冰箱分三步： 第一部分：提取 skills 这是基础。从小说里提取那些可以玩的“知识点”——可能是侦查技巧，可能是推理方法。 第二部分：做一个 MVP 有了 skills 之后，不是直接做成完整的游戏，而是先做一个最小可行性产品——能跑起来，能玩，哪怕很简单。 第三部分：强化互动性 这个是我最看重的。我不希望玩家只是“点点点”看剧情，而是要有探索感，能有延伸性。 技术框架怎么搭？ 我粗略想的是用 scenes 管理剧情场景，用 clues 管理线索收集，然后设计一个推理机制，让玩家能够主动运用 skills 去发现真相。 不过既然是游戏，还是时刻提醒自己： 别光想技术怎么实现，多想玩家怎么玩。 玩家需要什么样的探索感？如何让 skills 变成可玩而不仅仅是可看的工具？ Part 4: 踩坑 —— 贪心的代价：37 个技能的混乱 书我选了《拉普拉斯的魔女》。 PDF2skills 这个项目很给力，一口气帮我提炼了 37 个 Skills——密室杀人、不在场证明、时间诡计、气体动力学、心理诱导……看着这些 skills，我当时特兴奋： 这么多好东西，全塞进游戏里很期待其中的化学效果。 那时候我的想法很简单：skills 越多越好。 结果，飞是没飞起来，直接坠机了。 我一进入游戏，一脸懵逼：我是谁？我在哪？我要干什么？ 没有目标，没有引导，只有一堆不明觉厉的名词。整个游戏显得很乱。 这时候我才意识到一个问题： Skills 只是食材，不是大餐。 虽然提炼出了 skills，但拿这些 skills 来做什么，是我自己去定义的。 我想做的是侦探游戏，那我就需要给它一个定义——核心玩法是什么？需要哪些模块？这些都要想清楚。 我试图复刻之前的那个 Demo，但我发现， 不是 skills 越多越好，而是逻辑越顺越好。 要有线索、有探索感、有延伸性。 小说文本怎么处理？嫌疑人怎么设计？能不能对话？这些点其实是要慢慢用提示词去调整的，不是一步到位的。 所以我不得不忍痛割爱：砍！先挑几个 skills 出来做一个 MVP，看看可玩性，而不是一下子把 37 个全塞进去。 Part 5: 破局 —— 让玩家不再只是“看戏” 砍完技能做出的 MVP，还是差点意思。玩家知道凶手是谁，但不知道怎么推理出来的，这没意思。 我想推翻重来，但不知道该往哪个方向改。最后，我逼着自己设计了两个核心机制，这才把游戏盘活了： 机制一：戳穿谎言（打脸模式） 这不是简单的“我有线索”。 你必须先找到关键线索，然后你才能去找证人、嫌疑人，才能戳穿他们的谎言。 比如你发现了“红酒里有安眠药”这条关键线索，你再去找嫌疑人，他如果说“我什么都不知道”，你就可以直接打脸。因为我也没设计 AI 对话去盘问，所以是基于关键线索的触发——你找到了，才能质疑。【之前玩的 demo 是可以自己输入对话，质疑嫌疑人，我还没做这种设计】 机制二：推理引擎（脑补模式） 当你收集到多个线索，它们会自动形成一条推理链，告诉你这些线索能推理出什么。 比如： 线索 A：红酒里有安眠药 线索 B：备用钥匙不见了 线索 C：密室是从内部反锁的 A + B + C =??? 这三个线索组合起来，推理引擎会告诉你：凶手有备用钥匙，不需要密室手法，红酒里的安眠药是真正的凶器。 这样玩家就知道为什么他是凶手，或者凶手干了什么事。有了这条推理链，你才能更好地猜凶手是谁。 这两个设计让游戏从“看故事”变成了 “玩逻辑” ——不是瞎猜，而是有理有据地推理。 Part 6: 脑洞 —— 无限流与“想法即代码” 目前的版本完成了 60%，它还很简陋。但它的骨架（scenes + clues + reasoning）已经立住了，是个可以无限扩展的框架。 往后看，我有三个脑洞想填： 第一个是 接入真·大模型 。现在虽然是脚本对话，但未来可以让每个嫌疑人的对话都不一样，每次游戏都是新的体验。你甚至可以保存自己喜欢的游戏版本，分享给别人玩。 第二个是 无限案件生成 。我现在只用了几个 skills，后续可以把所有 37 个 skills 按照这种布局，做成多个案件——或者更疯狂一点，做成抽卡：玩家随机抽几个 skills, AI 重新生成剧情。那就比较像现在的 AI 生成项目了。 第三个是 宇宙化 。我自己就想着说，把《拉普拉斯》这本小说的所有技能揉成一个更大的、更丰富的无限流游戏。 回顾整个过程，我最享受的其实不是 Vibe coding 的那一刻，而是 思维实体化 的过程——从看书到提炼 skills，再到设计玩法，看着一个模糊的想法一点点变成屏幕上可交互的像素。 在这个过程中，我意识到一件事：别人的框架只是个参考，你完全可以在里面有自己的东西。我选了不同的书，设计了不同的框架，做出来的游戏也跟之前看到的不一样。 这个项目给我挺多延伸思路，我也希望给看到这篇文章的人一个启发： 你还能干什么？ 如果你喜欢侦探小说，可以做成推理游戏；如果你喜欢技能类的书——比如编程、烹饪——是不是也可以提炼成 skills，做成学习工具？如果你手里有大量的 PDF 文档，能不能也把它们变成可玩、可用的东西？ 它的延伸性非常强，希望你不仅仅是看到只是把侦探小说做成游戏。 在这个 Vibe coding 的时代， 代码已经不是门槛了，想法才是最稀缺的资产。
