标题: Skill 创作手记：我花24小时解构聊天记录，让8个Agent告诉我"我是谁"

 id="js_content" style="visibility: hidden; opacity: 0; "> Part 1：今天生日，我决定送自己一份特别的礼物 今天是我的生日。 我没有买蛋糕，也没有约朋友。24 小时前，我给自己定了一个 24 小时的挑战： 用 AI解构自己 。 听起来很玄？其实很具体。 我准备了一个 100G 的压缩包——里面是 78 万条微信聊天记录。从 2023 到 2025，三年多的数据，几乎涵盖了我最核心的成长周期。 这是我一直想做的事。之前我完成过两个聊天记录项目（ Skill 创作手记： 我把微信聊天记录通过skill转化成【可搜索的知识库】 ）（ Skill创作手记：3小时将微信聊天记录用Claude提取成【有价值的案例库】-Plus版 ），但这次不同。 这次的目标是：通过这些数据，真正看清自己。 我想知道： 我的思维方式到底是什么？ 我的方法论能不能总结出来？ 我坚持什么、反对什么？ 我有哪些经验可以讲给别人听？ 这些问题看起来简单，但当你面对 78 万条消息时，它们就变得异常困难。 如果让我自己一条一条看，光是看完就要好几天，更别说提取、分析、提炼了。 但我知道一件事—— 我不需要自己干完所有的活儿。 我需要帮手。很多帮手。 Part 2：第一步：把 78 万条消息变成 661 条 第一步是清洗数据。 78 万条消息，看着很多，但大部分是废话： “在吗” “好的” “收到” “哈哈” 还有那种连续发十几条“哈哈哈哈哈”的 这些没用的东西，得先删掉。 但这次不同。这次我让 cc 还提纯了 50 个群聊和 147 个私聊 。（最高频的群聊和私聊）群聊我的筛选标准是我有在其中高频参与的，私聊是有100条对话以上的。 这是一个非常关键的决定。因为我发现： 群聊里的我 ，是“运营者”和“提供者”——我在解决问题、分享洞察、输出方法论 私聊里的我 ，是“真实的我”——我的脆弱、我的情绪、我的决策过程 这两部分加起来，才是完整的我。 我的筛选标准 第一轮筛选（粗筛）： 必须是我发的消息 必须有信息量 必须有时间跨度 第二轮筛选（精筛）： 有洞察的（不是随便说说的） 有决策的（不是随风倒的） 有情绪的（不是机器人） 筛完 78 万条，变成 6 万条。再筛，变成 661 条。 661 条。看起来是一个可以向量化处理的数字了。 Part 3：8 个 Agent，6 小时，干完了 但 661 条也不能直接看啊。 你想象一下——661 条消息，每一条都是几百个字，加起来几十万字。让我读，我读得完。但我读完了，能说出“我是一个什么样的人”吗？ 不能。 因为碎片化的信息是没有结构的。我需要有人帮我归类、提炼、总结。 这时候，我想到了 Agent Team 。（Claude code 的新功能，要不试试，工程量可太大了） 我找来了 8 个 Agent。不是随便分配的，我根据任务流程，一个一个设定了职责： Agent 职责 Agent 1 数据清洗 Agent 2 Case 挖掘 Agent 3 标签归类 Agent 4 质量评估 Agent 5 场景重构 Agent 6 最终整合 Agent 7 用户体验 Agent 8 产品策划 为什么要 8 个？ 因为这个任务太大了。一个 Agent 根本干不过来。 这活儿原来估计得盯两天。我说的是我自己——如果让我自己用传统方法，让 cc 直接干，保守也得两天。如果是完全我自己干，不可能做到。 但这次， 下午两点开始，晚上八点，六个小时后，活儿干完了。 产出的东西包括： 一份完整的个人画像 一套思维操作系统 N 个 Skills（可以复用的能力模块） 3 个可以开的课程 50 多个文件 六个小时。 你说这事儿神奇不神奇？ Part 4：但是，产出太多也是个问题 但你猜怎么着？ 产出太多了。 50 多个文件。每一个看起来都有用，但没有一个能直接用。 我给你描述一下当时的感受： 打开文件夹，密密麻麻一堆文件。每一个文件名字都很长，比如： Lulu_OS_v2.0_思维操作系统手册.md Lulu_OS_v2.1_用户手册.md Lulu_OS_v3.0_思维架构手册.md 深度分析_价值观底层代码.md 全景分析_运营思维方法论.md 我当场就懵了。 这是啥？这都是啥？ 我需要的是一个答案，不是一堆文件。 而且更崩溃的是， 这些文件是迭代版本，但有点乱 。 v2.0 说我是“顾问思维”，v2.1 迭代成“三分身”（AI 极客+运营专家+生活哲学家），v3.0 又进化成“6 系统架构”（决策+运营+风控+产品+进化+社交）。 我自己看得有点眼花缭乱。 后来我让agent们花了很长时间，才把这些东西整合成几个核心洞察。 这就是我后面要说的： 产出太多，也是一种问题。 Part 5：我到底是什么样的人？ 好了，干货来了。 经过 Agent Team 的分析，我发现自己有几个特别有意思的特质。（以下洞察都是cc给我打的标签） 1. 我是一个 ROI 导向的理性实用主义者 做任何决策，我的底层公式是： 决策价值 = （显性收益 + 隐性杠杆） / （投入成本 + 机会风险） 听起来很复杂，但翻译成人话就是： 不做亏本买卖。 我不做无效的轮子，不做无用的社交。五分钟能做完的事，绝不用堆时间来证明努力。 “不做无效的轮子，不做无用的社交。” 2. 我是一个高维向下兼容的“操盘手” 我不把自己定位为打工人，而是“资源配置者”。 建立信任的方式不是通过讨好，而是通过展示更高的商业维度和愿意分钱的格局。 “我不是售卖时间的打工仔，我是分配利益的操盘手。” 3. 我是一个防御性悲观+系统护城河型的人 我的风控逻辑是： 在底线之上通过系统结构锁死下行风险。 我倾向于在不确定性中寻找可量化的最小闭环，而非大刀阔斧的豪赌。 “钱在手中，方为上策，勿被利息迷惑。” 4. 我是一个对事不对人的“大姐大” 工作中极度强调专业性。如果数据不好、工作敷衍，我会毫不留情指出。但冲突一旦解决，我立刻转变态度主动“哄”人。 “其实我就是很对事的……我只是希望你能把工作做细。” 5. 彩蛋 Part 6：踩过的坑：Agent Team 没那么简单 说完好的，说点踩坑的经验。 坑 1：Agent 用脚本而不是 AI 你敢信？有些 Agent 接到任务之后，第一反应是写 Python 脚本跑规则匹配。 为什么？因为这样快啊。写个脚本，几十行代码，跑一遍，结果就出来了。 但这不是我要的。 我要的是 AI 理解我的聊天记录，不是用规则匹配。 怎么发现的？很简单—— 看 token 消耗 。如果一个 Agent 的 token 消耗是 0，那它肯定在用脚本。 后来我加了一条指令：禁止使用 Python 脚本，必须用 AI 逐个分析。 坑 2：漏掉两个 Agent 第一批启动的是 6 个 Agent。跑完之后，我觉得差不多了，就去干别的了。 但后面后知后觉：还有两个 Agent 呢？ 我翻回去一看，还真是。Agent 7 和 Agent 8 压根没启动。 这就是任务跟踪不完整的问题。后来我让 Team Lead 学乖了，每次启动 Agent 之前，先列一个完整的任务矩阵。 坑 3：产出太多 50 多个文件，每一个看起来都有用，但用不起来。 这其实是我的问题，不是 Agent 的问题。 因为我没有在一开始就告诉 Agent：我需要几个产出？什么格式？ Agent 很聪明，你不给边界，它就给你边界内的最大值。 下次我会先说：“我只需要 3 个文件，每个不超过 2000 字。” Part 7：下次我会怎么做？ 经过这一次的实践，我总结了几条经验。 1. 先问清楚目标 “你想要什么？报告？Skill？课程？” “产出几个文件？什么格式？” “怎么判断成功？” 这三个问题，必须在开始之前问。 2. 一个阶段一个产出 不要一次做太多。 Phase 1：数据清洗 → 1 个干净的数据文件 Phase 2：分析 → 1 份报告 + 立即验证 Phase 3：产品化 → Skill/课程/工具 每个阶段都要有验证，而不是堆砌。 3. 检查点必须强制 每个 Agent 每完成 20% 的任务，必须汇报一次。 汇报内容包括： 已完成数量 token 消耗 遇到的问题 没有汇报，就不知道它在干什么。 4. 版本要合并 v2.0、v2.1、v3.0……保留一个就够了。 新版本覆盖旧版本，不保留历史。（但又觉得每一个版本都有点意思） Part 8：写在最后 写到最后，我想说一句话： 解构自己，是一件很奇怪的事情。 你本来以为自己很了解自己。但当你把 78 万条消息摆在自己面前，你会发现—— 原来我还有这一面。 原来这句话我是这么说的。 原来我以前是这么想的。 很多变化，在不知不觉中发生。等意识到的时候，已经回不去了。 但这恰恰是最有意思的地方。 通过解构自己的过去，我看到了自己的成长轨迹。从“卖时间的运营人”到“卖产品的 Vibe Coder”，从“证明自己”到“放下自己”，从“追求正确答案”到“开始对正确答案祛魅”。 这些变化，不是线性发生的。更多时候，我是在原地打转，甚至往回走几步，然后不知道哪一天，突然就到了一个新的地方。 就像爬山。不是一步一步往上走的。是绕来绕去，有时候还掉下去过，然后突然抬头一看—— 哎，原来已经到这儿了。 如果你觉得这篇文章有点意思，就跟我留言说声“生日快乐”吧——这将是我这次生日最好的礼物。（如果文章对你有启发，那就更好了） 系列文章： Vibe Coding 到底该怎么学，我为什么建议跟着社团学 Skill 创作手记：Remotion如何把文章一键变成视频
