我最近折腾了一套“AI 助手上岗”的工作流：用 OpenClaw 接入飞书，让它定时发日报、监控消息、自动化一些流程。刚开始体验非常丝滑：飞书消息收发稳定、组织内沟通也方便，一切看起来都像“这就是我想要的生产力操作系统”。
直到它突然给了我一闷棍：自建应用的 API 调用次数到 10000 之后，直接限额。
一、1 万次/月听起来不少？但对 Agent 来说不够用
你可能会说：1 万/月听起来不少啊？
但把它换算成“Agent 的日常行为”，就很吓人：
1）你开了一个每天 7:00 的定时任务：一次拉取、一次整理、一次发送，背后可能是多次 API 调用。
2）你让它监听群聊：拉历史、看新消息、查成员、回消息……只要你认真用，它就会稳定消耗额度。
3）更致命的是调试期：今天改一次配置、明天多加一个提醒，反复测试会让调用量直线上升。
结果就是：这个额度基本用几天就没了。尤其是你真的把它当“工作助理”在跑的时候，消耗速度只会更快。
二、最难受的不是“慢一点”，而是“直接断供”
更关键的是：它限额之后，不是“慢一点”，而是“直接拒绝请求”。
也就是说，你的助理会突然失声：该发的日报不发了，该回的消息回不了，整个链路断掉。你以为自己在“养一个能长期工作的助理”，结果变成了“养一个随时可能失联的玩具”。
三、飞书的解决方案：升级（我当场就无语了）
然后飞书给你的解决方案也很直白：升级。
我能理解商业产品要收费，但把“开放平台 API 调用次数”卡在这么低的水平，对 Agent 场景来说就是硬伤——你不是在限制滥用，你是在限制正常使用。
我是在做自动化，不是在做“氪金版沟通工具”。
四、我最后怎么解：迁到 Discord，立刻踏实
所以我干脆换了：把 OpenClaw 的主要投递和交互迁到了 Discord。
迁移之后最大的感受是：心里踏实。
1）该跑的定时任务照跑。
2）消息能稳定送达。
3）不用每天盯着“还剩多少额度”，也不用担心突然被“掐断供氧”。
五、给准备接入的人一个建议
如果你也打算用 OpenClaw（或者任何会高频调用平台接口的 Agent 框架）来做长期运行的助理，我的建议很明确：
1）别把飞书当作首选“Agent 入口”。飞书适合人类协作，但它对“高频自动化/Agent”不友好，尤其是自建应用的 1 万次/月限额，会在你刚跑顺的时候突然把你按停。
2）如果你必须用飞书：要么一开始就接受升级成本，要么从架构上降低调用（减少轮询、合并请求、降低心跳频率、减少读取类 API）。但说实话，这些会直接削弱 Agent 的体验——你为了省额度，反而把“助理”变成“半残”。
3）更现实的选择：换一个更适配 Agent 的沟通/投递渠道。像 Discord 这种以 Bot 生态为核心的地方，天然就把“自动化消息”当成正常用户，而不是当成需要被限制的异类。
总结一句话：飞书接 OpenClaw，前两天很爽；第三天开始你就会问自己——我是在训练助理，还是在训练自己忍受限额？
