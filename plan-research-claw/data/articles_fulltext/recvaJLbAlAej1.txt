# 折腾 Clawdbot 后，我换成了更简便的 memU Bot

最近关于 Clawdbot 的讨论很多，我不太想只看演示，还是习惯自己把东西跑一遍。
结果配置过程并不轻松。
环境、权限、依赖、配置，一个都不能少。
教程写得还算清楚，但默认你知道自己在干什么，如果没有工程背景，基本不可能一次装好，而且即便让模型自动帮你去安装这个clawdbot，也还得看看你用的是啥模型。。。
所以折腾到一半的时候，我其实已经有一个判断：clawdbot 这套东西工程上很厉害，但它更像是一个能力集合，而
不是一个默认会被普通人长期使用的产品。
未来应该有更强的产品以及更轻便的产品形态出现。
也是在这个背景下，我转而装了 「memU Bot」，那个 “主动式 AI”“7×24 跑任务” 的 Agent 方案。
我只需要配置完成之后，你直接就可以在飞书呼唤它，然后它就能够随时待命，这是我刚刚问它是否还在工作的回应。
下载地址：https://memu.bot/
memU Bot安装过程很简单，直接就下载客户端安装包，自己只需要配置好模型的 API Key，接一个通讯工具就能跑起来。
通信工具我用的是飞书，配置下整个过程几分钟搞定，没有环境折腾。
咱们直接看这个官方的搭建教程文档，总共也就5、6步骤，基本上可以丝滑上手，参考地址：https://memu.bot/tutorial/feishu
很多人问这种 AI Agent 搭建完成之后，究竟有啥用，然后 memU Bot 思路不一样的，不是“装得快”，而是后面的使用方式。
我先从最基础的事情开始试。
我只丢了一条微信文章链接给它，没有说阅读框架，也没有交代输出形式，只说了一句“这个文章讲了什么”。
它给我的不是那种通用摘要，而是明显偏向我平时关注的产品判断和方法论视角。
一些不重要的内容被自然忽略掉了。
关键点在于，它没有再问我“你想怎么看”，这意味着它不是每次从零开始理解，而是已经把我之前使用它的方式记了下来。
这个体验如果没有 memory，是很难成立的，这是因为memU这个开源项目之前本身就是做记忆存储管理的，我只需要把之前的一些记忆数据通过平台的 api key 进行关联，就可以让这个 bot 更理解我的特点。
接下来我开始测试它的持续运行能力。
然后我就又是一句顺口的话，让它把上面的内容直接做成word文档。
制作成文档后先是给了我存储的位置，然后方便我后续去检查里面的内容，这里面就直接到把公众号的文章内容分析情况给存储到一个文档里面。
然后又是顺便直接帮我做成一份PPT文档，我在发起这个诉求之后，大概过了一段时间，它完成任务之后又是主动的回复了信息，告诉我说，
“老板你可以前往查阅了。”
于是乎，我们就能得到这样的一个快速生成的ppt文件初稿。
同样的，晚上刷 B 站的时候看到一个视频，感觉内容可能有用，但当时不想看。
我直接在手机端继续把链接丢给 memU Bot，只说了一句“提取一下这个视频链接”，然后就关了电脑。
过一会儿再看，飞书机器人就发来一条信息，告诉我视频已经提取完成了。
视频已经被下载到本地，内容转成了文字，mp4、json 视频结构文件、jpg 封面图都已经给的清清楚楚。
我没有在线等待它执行，也没有中途确认。
这一步对我来说很重要，因为它验证的是：
我不在的时候，它会不会继续跑。
很多 AI 工具一旦对话中断，就停在那里等你。
memU Bot 不是，它是一直在后台跑着的，只要任务没结束，就会继续。
到这里为止，其实已经能看出 memU Bot 的一个核心取向：
它并非把所有能力一股脑塞进来，它做是把“长期运行 + 记忆”作为底座，其它能力靠扩展。
这时候 Skills 和 MCP 的设计就开始显得很关键。
memU Bot 本身装完之后，能力其实是比较克制的。
很多事情并不是默认就会做，而是需要你自己去装合适的 Skills。
我点开这里面的 GitHub 搜索，让它自己告诉我当前已经装了哪些 Skill，我们也可以直接向他发起提问我们现在有的 Skills 都有啥，在手机端就能知道整个电脑后台里面的基础工具能力配置。
它会很清楚地列出来，并告诉你每个 Skill 是干嘛的。
如果你想要新的能力，可以直接让它帮你找。
比如我当时需要做阅读和整理，就让它去找一个能生成 Skill 的 Skill，装好之后，再用它来做我自己的定制。
整个过程不是写代码，而是“用对话配置能力”。咱们在做的事情，本质上是搭自己的能力组合，而不是学习一套复杂的系统。
MCP 也是类似工具的逻辑。
它并不是内置一堆你用不到的服务，也是那种你需要的时候自己去装。
装完之后，它就可以在任务里调用。
这意味着什么？意味着你给它是一个可以随着你需求变化不断扩展的执行系统。
更重要的是，
这种设计非常适合 7×24 跑。
因为能力是你自己装的，它只会在需要的时候调用，不会乱用，从工程角度看，这是一种非常务实的取舍。
它没有试图在一开始就覆盖所有场景，它把
“
能不能长期稳定跑
”
这件事放在第一位。能力不够，就加，不用的能力，就不装。
这也是我觉得 memU 整个项目在增长上会很快的原因。
傻瓜式部署，让更多人能跑起来，开源底层，让有能力的人愿意深度参与。
Skills 和 MCP 的扩展方式，又给了足够的空间去形成生态。
如果你只是想研究技术，Clawdbot 的工程复杂度和可玩性确实更高。
但如果你的目标是让一个 AI 助手长期跑在你电脑上，替你干活，而不是让你反过来伺候它，那 memU Bot 的取向显然更现实。
对我来说，这次测试最大的价值不在于某一个功能，而在于我已经开始把一些事情放心地交给它
7×24
去跑。
当然，如果你玩的花一点的话，还有一些有趣的玩法，你可以自定义它的模型提示，比如像我这样，让它称呼我为海总。
于是呢，它在每一次任务完成后，都会亲切的称呼你一声“海总...xxx” 😆
哈哈哈哈哈....快去玩玩看吧！
