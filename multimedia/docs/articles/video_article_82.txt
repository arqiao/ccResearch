标题: Skill 创作手记：Remotion如何把文章一键变成视频

正文:

Part 1：为什么我要做这个 skills
那天我突然意识到一个问题——
我写了 4 篇文章，但每次想做成视频分享的时候，总觉得卡点很多。
你要录音、剪辑、调格式、加字幕……一套流程下来，半天时间就没了。而且最烦的是，直接拍自己没有感觉，做动画视频配音要卡点，如果没录好，音频就得重新录。关键是我剪映真的用的一般。
我就想，能不能自动化？
其实上期项目做完“侦探游戏”的时候，我就有了这个想法。当时那个只是录屏，不是正式的视频，但我觉得它更适合做成视频——有画面、有声音、有节奏。所以这次我想做一件事：
把整个流程标准化。
不过说实话，这次做出来的 Project 2 也不算完美。因为我还没有嵌入游戏录屏演示的部分——就是真正展示“这个工具是怎么用”的那个画面。但后面可以再加，完成比完美更重要。
Part 2：完整工作流是怎么演进的
一开始我是看了黑客松一个同学的项目分享，他用 Remotion 把文章转成视频加了配音，给了我很大启发。然后我又去别的公众号大概摸了一下流程，看看别人是怎么做的。
但我发现，别人的流程不一定适合我。
我的核心需求
我的核心需求是：写完文章就能出视频。
希望做这件事，还是我对私域的理解上，我希望补足视频这环：
视频是更大的流量词，它可以让更多人看见你。但视频很短，别人看了可能觉得“有点意思”，但不足以建立信任。
公众号文章是更深度的内容，它可以让别人信任你——看到你的思考过程、你的专业能力。
所以我想要打一个闭环：
视频引流 → 文章沉淀 → 再引流到私域。
其实我做私域顾问的时候，给别人的建议都是这样的。但我觉得，给别人建议，自己也要践行，这样才叫知行合一。
6 个阶段的完整流程
所以我需要一套能快速、稳定地把文章变成视频的流程。按自己的需求，我分成了 6 个阶段：
阶段 1：项目初始化
我给一个文件夹（包含 TXT 文稿 + 图片素材），cc读取并确认内容。
阶段 2：内容策划
建议缩减到多少秒（45-60 秒之间），撰写逐字稿第一版，计算预估时长。
阶段 3：视频开发（关键）
设计 6 个场景的视觉方案，开发场景组件，浏览器预览确认。视觉效果没问题了再进入下一步。
阶段 4：音频生成
准备 JSON 配置文件，调用 克隆音色的 API 生成 6 段音频。
阶段 5：音视频同步
第二轮逐字稿确认（可以调整语气、时长、倍速），同步音频到视频时间轴。
阶段 6：多版本输出
优先渲染黑金版 3:4，确认后批量渲染其他版本。
💡 这套流程跑下来，如果熟练的话，半小时内能调出来。而且效果比一般的 PPT 好很多。
Part 3：踩过的坑（从 6 小时到 1 小时）
一开始我以为 4 个小时就能把整个流程跑通。结果第一轮做下来，花了 6 个小时。
最花时间的是配音生成——不是我自己录音，而是用 TTS 生成时遇到了各种奇怪的问题。
一开始选 TTS 服务就很纠结。我看了 ListenHub，效果确实好，但太贵了，要 20 美金才能试用 API。后来想了想，还是时间性价比优先，选了 MiniMax。整体下来，一个视频的配音只要 3 毛钱，还是挺划算的。
但选完之后，问题才刚开始：
坑 1：音频叠加混乱
我第一次生成完 6 段音频，加到视频里，发现所有声音叠在一起了，像大合唱一样，完全听不清在说什么。
我让 cc 查了很久的文档，才发现 Remotion 的 Audio 组件如果不用 Sequence 包起来，会全部同时播放。解决方法也很简单——用 Sequence 组件隔离时间轴。
但这个问题让我明白：
流程顺序很重要。如果视频没有确认，先生成音频，最后做视频流的时候，声音就得硬塞进去。
坑 2：文件夹定义混乱
这个问题是第二次复用流程的时候才发现的。
第一次做 Project 1（侦探游戏）的时候，我没有把文件夹结构定义清楚——图片素材、音频文件、配置文件都混在一起。
第二次做 Project 2（聊天记录变知识库）的时候，生成的时候出现了混乱——不知道用哪个项目的素材，也不知道输出到哪里。
我才意识到：环境隔离很重要。每个 project 应该有独立的文件夹，分环境管理，避免素材错乱。
坑 3：时长超了
也是在 Project 2，我第一版逐字稿写完，算了一下字数：288 字。按照舒适语速（4-5 字/秒），需要 62 秒左右。但我原始时间轴只分配了 45 秒。
怎么办？我不想删减内容，所以我选择调整时间轴——动态分配场景时长，允许总时长浮动到 66 秒。个别场景如果还是超，就用 1.2 倍速解决。
这样既保证了内容完整，又保持了表达自然。
做完第一轮，我不太放心，又跑了一轮验证。这次快了很多，大概 3 个小时。
但这两轮下来，我把所有经验都梳理完了（总共 22 个关键经验）。现在理论上可以把流程缩短到1小时内——虽然还没实测，但已经挺顺的了。
Part 4：视觉设计经验
视觉设计这块，其实没有标准答案。
我这次用的是 票据风——登机牌布局、功能性指示符（ @ CHECK IN、™）、粗边框 + 厚重阴影。场景 1 和场景 6 首尾呼应，创造视觉闭环。
这是我个人比较喜欢的风格，有设计感，又不抢戏。跟我公众号文章一致。
每个人可以定义自己的风格
但我觉得，每个人可以定义一套自己喜欢的风格。
有的人可能喜欢极简风，有的人可能喜欢赛博朋克，有的人可能喜欢复古风。关键是你有一套自己的视觉语言——配色、字体、排版、动画效果——这样所有视频统一在一起，才能形成个人品牌。
而且 Remotion 的组件化特性，让你可以很方便地定义和复用这些设计元素。
比如 Logo，你可以定义一次，然后所有场景都用同一个。比如前景图，你可以定义一套排版规则（圆形、斜切、六边形），然后根据场景需要灵活选择。
一个重要原则
但有一个原则我觉得很重要：前景图不要太多。
我这次只保留了 2-3 个前景图，避免过于杂乱。位置控制在top: 10%，不压到底，留出呼吸空间。
动画效果也是一样——聊天气泡、进度条、饼图这些动态效果，可以增加视觉层次，但不要为了动画而动画，要服务于内容。
这是第二轮验证的视频，效果我觉得70分还是有的。
💡 所以视觉设计的核心不是“怎么做漂亮”，而是“怎么做统一”——统一的风格、统一的规则、统一的个人品牌。

Part 5：可复用的经验（思路 > 技术）
先分享我整体用的是 github 的 remotion skills 和 minimax 的 api 克隆音色
这套流程跑下来，我觉得最核心的经验不是技术细节，而是思路。
1. 流程一定要搞清楚
这不是一个技术问题，而是一个检查点问题。
我的核心需求是：整体效果和内容基本匹配。
我希望能检查——视频效果怎么样？音频节奏对不对？逐字稿自然吗？
如果流程顺序不对，你可能跳过检查，直接往下做。等到后面才发现不对，返工反而更花时间。
但我也不是说每一步都要踩得很细。关键是在正确的地方检查——视频效果确认后，再生成音频；音频生成后，再同步时间轴。
2. 两轮逐字稿确认
第一轮
在生成场景代码前，确认场景内容和时长分配。
第二轮
在生成音频后，调整语气、节奏、个人风格。
两步走，比一步到位更精准。而且每一轮都有具体的确认点，不会越跑越偏。
3. 视频开发在音频生成之前
这是我最深刻的体会。
如果你先生成音频，最后做视频的时候，声音就得硬塞进去。有些内容你想放进去，但放不进去——因为它不像视频剪辑那么方便。
正确的顺序是：先做视频开发，浏览器预览确认视觉效果，然后再生成音频。音频可以配合视频调整，但视频很难配合音频。
4. 环境隔离
每个 project 独立文件夹（TXT + 图片 + 音频），分环境管理，避免素材错乱。这个看似简单，但第二次复用流程的时候才会发现多重要。
⚡ 这些经验，本质上是 思路 > 技术。技术会过时，但思路永存。
Part 6：Skills 的本质：解决问题，而非编程
这套流程做完，我觉得最大的价值不是“能自动生成视频”，而是“能相对快速生成视频"。
因为快，所以视频和文章可以一起出——文章写完，视频也出来了。
其实花时间本身我倒没这么在意，1 个小时也不短，在我可以接受的范围，关键是流程可复用，质量可控制，不是纯抽卡，我才愿意多投入时间。
双向导流的私域闭环
视频和文章一起出，就能形成私域闭环：
视频是流量入口，让更多人看见你；
文章是信任沉淀，让别人更了解你。两者结合，双向导流。
Skills 的本质：想象力即代码
做完这个项目，我对 Skills 有了更深的理解。
Skills 不是为了编程而编程，而是为了解决实际问题。
你不需要懂 Remotion 原理，不需要会调 TTS API。你只需要说清楚需求——“我想把文章变成视频”，Skills 帮你实现。
这才是 想象力即代码：任何工作流，只要逻辑清晰、步骤明确，都可以变成可复用的 Skill。今天是文章转视频，明天可以是数据分析、内容整理、素材生成……
编程不再是门槛，想象力才是。
